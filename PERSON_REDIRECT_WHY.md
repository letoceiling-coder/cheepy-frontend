# Почему /person может редиректить на /auth

## В коде редиректа при загрузке /person нет

- Маршрут `/person` в **App.tsx** рендерит **PersonLayout** без какого‑либо guard (строки 147–165).
- В **PersonLayout** нет `useEffect` с `navigate("/auth")`; единственный `useEffect` только анимирует контент.
- **`<Navigate to="/auth">`** в проекте **нигде не используется**. Есть только **`<Navigate to="/admin/login">`** в AdminAuthGuard для админки.
- Все вызовы **`navigate("/auth")`** — только в **обработчиках клика** (кнопки «Войти» в шапке, сайдбаре, модалке). Они не выполняются при открытии страницы.

То есть при открытии `/person` текущий код **не делает** автоматический редирект на `/auth`.

---

## Откуда тогда редирект в браузере

Если при переходе на `https://siteaacess.store/person` всё равно уходит на `/auth`, это почти наверняка **старый код в браузере**:

1. **Кэш JS**  
   Браузер отдаёт старый `index-*.js` (например, с бывшим guard’ом или редиректом). Сервер уже отдаёт новый бандл, но браузер его не запрашивает.

2. **Service Worker**  
   Раньше мог быть зарегистрирован SW (или от другого билда). Он может отдавать закэшированный старый HTML/JS, в котором есть редирект.

3. **Жёсткий кэш (CDN/прокси)**  
   Если перед сервером стоит CDN или прокси с кэшем, они могут отдавать старую версию страницы или скриптов.

---

## Что сделать по шагам

### 1. Убедиться, что грузится свежий бандл

- Открой **https://siteaacess.store/person**.
- **F12** → вкладка **Network** → включи **Disable cache**.
- Обнови страницу (**Ctrl+Shift+R** или Cmd+Shift+R).
- В списке запросов найди основной скрипт вида **`index-XXXXXX.js`**.
- На сервере выполни:
  ```bash
  grep -oE 'index-[^.]+\.js' /var/www/siteaacess.store/dist/index.html
  ```
  Имя файла в Network должно **совпадать** с выводом этой команды. Если совпадает — браузер уже тянет актуальный билд; если нет — грузится старый (кэш или другой источник).

### 2. Отключить Service Worker

- **F12** → вкладка **Application** (Chrome) / **Storage** (Firefox).
- Слева **Service Workers**.
- Если для `https://siteaacess.store` есть активный worker — нажми **Unregister**.
- Обнови страницу и снова зайди на `/person`.

### 3. Проверить в режиме инкогнито

- Открой **новое окно инкогнито** (без расширений и без старого кэша).
- Перейди на **https://siteaacess.store/person**.
- Если в инкогнито редиректа нет — значит проблема в кэше или SW в обычном профиле.

### 4. При необходимости пересобрать фронт на сервере

Чтобы у бандла точно сменилось имя (новый хэш) и браузеры подтянули новый файл:

```bash
ssh root@85.117.235.93 "cd /var/www/siteaacess.store && git fetch origin && git reset --hard origin/main && npm run build"
```

После этого снова проверь имя бандла на сервере (`grep -oE 'index-[^.]+\.js' .../dist/index.html`) и в Network в браузере.

---

## Итог

- **В коде** автоматического редиректа с `/person` на `/auth` **нет**.
- Редирект в браузере идёт из **старого закэшированного JS** или **Service Worker**.
- Действия: отключить кэш в DevTools, снять Service Worker, проверить в инкогнито, при необходимости пересобрать фронт на сервере и убедиться, что в Network грузится тот же `index-*.js`, что и в `dist/index.html`.
